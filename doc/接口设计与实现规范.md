# 接口设计与实现规范

## 1. 目的

本文规定所有 HTTP、应用服务、Repository、Gateway、Runner 和 Adapter 接口的统一写法。接口文件只允许包含类型、Protocol、抽象签名、常量和注释，不允许连接数据库、访问网络、执行 Git/Docker 或调用模型。

## 2. 三层接口

每个业务模块按以下顺序实现：

```text
router.py（HTTP 契约）
  → service.py（业务用例契约）
    → domain.py（状态与业务规则）
    → repository.py（持久化契约）
    → gateway Protocol（外部副作用契约）
```

- `schemas.py` 被 Router 和 Service 共用，定义请求、响应、查询和命令结果；
- Router 只完成认证上下文取得、参数解析、调用 Service 和状态码映射；
- Service 决定校验顺序、事务边界、状态变化、事件和幂等性；
- Repository 只保存或查询本模块数据，不编排跨模块流程；
- Gateway 是模型、Tool、Secret、GitHub、Git、外网检索等副作用的唯一入口。

## 3. 每个接口注释的必填内容

接口方法的 docstring 必须回答：

1. 谁调用它；
2. 每个输入从哪里来、是否可信、必填条件是什么；
3. 成功时输出什么，输出中的 ID 或版本代表什么；
4. 实现时按什么顺序处理；
5. 会修改哪些状态，会产生哪些事件；
6. 可能出现哪些业务错误，调用方能否重试；
7. 写操作如何使用 `idempotency_key`，并发更新如何使用 `expected_version`；
8. 是否接触密钥、外网、文件、Docker、Git 或模型，以及安全边界在哪里。

## 4. 公共规则

- ID 在传输层统一使用字符串形式 UUID；
- 时间使用带时区的 ISO 8601 UTC 字符串；
- 金额使用十进制字符串和三位货币代码；
- 写接口必须带幂等键；相同键和相同请求返回第一次结果，相同键和不同请求返回冲突；
- 更新已存在资源时必须检查 `expected_version`；
- 列表采用游标分页，默认 20，最大 100；
- HTTP 成功创建返回 201，查询返回 200，异步受理返回 202，无正文成功返回 204；
- 参数错误返回 400，未认证 401，无权限 403，不存在 404，版本/幂等冲突 409，Schema 不通过 422，限额 429，外部依赖失败 502/503；
- API 响应不得包含 API Key、Token、Secret 密文、宿主机绝对路径和未脱敏模型输入；
- 跨节点禁止新增内部 HTTP API；Git 事件必须校验节点 Git 身份、Schema、event_id、control commit 和 assignment epoch；
- 对外错误使用稳定 `error_code`，不得把异常堆栈直接返回客户端。

## 5. 类型所有权

- 浏览器与 Server：以 OpenAPI 为源，生成 `contracts_ts`；
- 中央 Scheduler 与节点：以 `maf_contracts.coordination`、`.maf/schemas` 和 Git 分支协议为源；
- 单模块请求响应：由该模块 `schemas.py` 所有；
- 跨模块领域原语：由 `maf_domain` 所有；
- 供应商特有字段：只能存在于对应 Adapter 内，进入 Gateway 前后都要归一化。

## 6. 实现步骤模板

初级开发者实现任意写接口时按此顺序：

1. Router 解析路径、查询、正文和请求头；
2. 认证中间件提供 `ActorContext`；
3. 校验格式、必填项和允许值；
4. Service 检查权限、资源存在性和版本；
5. 查询幂等记录，必要时直接返回历史结果；
6. 进入 UnitOfWork；
7. 调用 Domain 完成业务状态转换；
8. 通过 Repository 保存聚合与关联记录；
9. 同一事务写 Domain Event/Outbox；
10. 提交事务后才执行可延后的外部动作；
11. 返回不含敏感信息的 DTO；
12. 记录 trace ID、actor、耗时和结果，禁止记录密钥。
