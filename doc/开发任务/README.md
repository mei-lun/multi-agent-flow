# Multi Agent Flow 开发任务库

每个任务尽量控制在 0.5～2 个开发日，只解决一个主要问题。开发前必须先完成“前置条件”，开发后逐项满足“验收标准”。任务状态由后续 Git coordination 协议维护；本目录当前是实施基线。

## 执行规则

1. 按任务 ID 和依赖推进，不因目录顺序跳过前置条件；
2. 一个任务使用一个分支和一个 PR；
3. 不在当前任务中顺手实现“不包含”内容；
4. 公共接口需要变化时先修改设计和接口契约；
5. 每个验收项都应有测试、命令输出或文档证据；
6. 跨机器协作只使用 Git control/node/task 分支，不新增节点 HTTP。

## 改动范围规则

每个任务文档必须包含“允许改动文件”“允许新增或修改接口”“禁止改动”三部分：

- 只允许修改明确列出的生产文件或目录；测试文件也必须明确列出；
- `允许新增` 表示文件当前可以不存在，实现任务时按给定路径创建；
- 接口列表是该任务唯一可以新增或改变的函数、Protocol、DTO、API 或 Git Schema；
- 未列出的公共接口只允许调用，不允许改变签名、字段、状态语义或路径；
- 格式化工具产生的同文件机械变化允许保留，但不得借机修改其他模块；
- 如果范围不足，先更新任务文档并评审，或提交 `CodeStructureChangeRequest`；
- 每个任务自己的 Markdown 状态、证据和验收勾选始终允许更新。
